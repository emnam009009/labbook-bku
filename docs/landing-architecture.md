# Labyra Landing — Architecture Reference

> Living document for the **Labyra marketing site** at `labyra.com`.
> Implementation lives in a separate repo: `labyra-landing`.
> This file documents the architecture for cross-reference from
> the main `labbook-bku` repo. Last updated: R143d (May 10, 2026).

## Why a separate repo?

- **Independent deploy cadence** — landing changes daily during marketing pushes,
  app deploys are lower frequency and gated by tests.
- **Different stack** — Astro static SSG vs Vite SPA. Different `package.json`,
  different build pipeline. Mixing them increases CI time and dependency surface.
- **Different domain** — `labyra.com` (landing) vs `app.labyra.com` (product).
  Each has its own Firebase Hosting target.
- **Smaller contributor surface** — landing can be touched by marketers/designers
  without exposing them to the full app codebase.

## Stack choice

- **Astro 5** — static site generator, zero-JS by default. Fastest path to a
  Lighthouse 100 marketing site. Loads React islands only where interactivity
  is needed (waitlist form, chat demo).
- **Tailwind CSS 3** — same utility classes as the main app, easy mental
  model switching for the team.
- **TypeScript strict** — same standard as functions backend.
- **Zod** — schema validation for both Content Collections and form input.
- **React 19** — used only for interactive islands inside Astro pages.
- **Playwright** — E2E tests for landing flows.
- **Firebase Hosting** — same provider as app, share GCP project for billing
  consolidation (different hosting target per site).

Alternatives considered:
- **Next.js** — overkill for a static marketing site. Heavier bundle, longer
  build, more JS shipped to clients than necessary.
- **Plain Vite + React** — works but no built-in MDX/Content Collections,
  no built-in sitemap generation, more manual config.
- **Webflow / Framer** — fast initial setup, but locked into their hosting
  and pricing. Cannot integrate with Firebase Auth for waitlist gating later.

## Conventions

### Components vs Sections

- **Sections** are large, single-use blocks composed in pages. Each section
  is a complete narrative chunk (Hero, Features, Pricing). They typically
  consume data from `content/data/*.json` and render their own layout.
- **Components** are small, reusable primitives (Button, Badge, Card).
  They are presentational and accept props. Used by sections and pages.

### React islands

Default to Astro components (zero JS). Use React only when the component
**must** maintain client state across user interactions (form validation,
chat typing animation, theme toggle). Place in `components/react/`.

Hydration directives:
- `client:idle` — for forms below the fold
- `client:visible` — for animations triggered by scroll
- `client:load` — only when truly above-the-fold interactive (rare)

### Content Collections

All structured content (changelog entries, blog posts) goes in
`src/content/` with Zod schemas in `config.ts`. Static data (features,
pricing tiers, FAQs) goes in `src/content/data/*.json` for non-MDX
content that doesn't need full collection APIs.

### Styling

- **Tailwind utility classes only** — no separate `.css` modules per component.
- **Tokens in `tokens.css`** — CSS variables for colors, spacing, fonts.
  Tailwind config references these.
- **No inline styles** in components except for one-off dynamic values.
- **Dark mode** — not implemented in v0.1, can add via `class="dark"` strategy
  with `darkMode: 'class'` in Tailwind config when needed.

### i18n strategy

Phase 1 (current): `src/i18n/en.json` + `vi.json`, manual key lookup in
components via simple helper. EN default, VN toggleable.

Phase 2 (when adding JP/KR): migrate to `astro-i18next` or
`pages/[locale]/index.astro` pattern. Refactor scope: ~1-2 days.

## Build + deploy

### Local dev

```bash
pnpm install
pnpm dev          # http://localhost:4321
```

### Production build

```bash
pnpm build        # outputs to dist/
pnpm preview      # local preview of dist
```

### Deploy

Auto-deploys to Firebase Hosting on push to `main` via
`.github/workflows/deploy.yml`. Requires GitHub secret
`FIREBASE_SERVICE_ACCOUNT` and project ID `labyra-landing`.

Manual deploy:
```bash
pnpm build
firebase deploy --only hosting
```

### Cache strategy (firebase.json)

- `/assets/**` — `max-age=31536000, immutable` (fingerprinted)
- `*.html`, `*.json` — `max-age=300, must-revalidate`

## Numbering convention

Landing iterations are **completely decoupled** from app rounds. The two
repos run independent cadences and use different prefixes:

| Repo | Prefix | Format | Example |
|---|---|---|---|
| `labbook-bku` (app) | `R` | numerical, monotonic | R143, R144, R145... |
| `labyra-landing` (landing) | `L` | numerical, monotonic | L0, L1, L2... |

### Why decoupled?

1. **Different cadence.** Landing changes daily during marketing pushes
   (typo fixes, copy A/B, image swaps). App rounds change weekly with
   stress tests. Forcing shared numbers blocks landing velocity.
2. **Different reviewers.** Landing PRs may be reviewed by a designer or
   marketer with no app context. Asking them to know "we're at R157" is
   noise. App PRs need engineering review and should not be gated on
   landing iterations.
3. **Different release standards.** Landing deploys auto on merge-to-main.
   App deploys are gated by stress tests, manual sign-off, and may pin
   to specific Cloud Function regions.
4. **Cleaner changelogs.** Each repo has a self-contained CHANGELOG that
   reads as a coherent story rather than `[app]`/`[landing]` interleaved.

### Patch scripts vs direct commits

| Repo | Mechanism | Why |
|---|---|---|
| `labbook-bku` | Python patch scripts (`round-NNN-xxx.py`) | Legacy TS migration, idempotency, atomic multi-file edits |
| `labyra-landing` | Direct `git commit` | Single-file changes, no legacy debt, fast iteration |

Landing changes are usually small and scoped to one or two files
(`Hero.astro`, `pricing.json`). The atomicity benefit of patch scripts
does not justify the overhead. Exception: if a cross-cutting change
hits 10+ files (rebrand color tokens, swap font family across all
sections), a one-off patch script is acceptable but optional.

### CHANGELOG entry format

**App (`labbook-bku`):**
```markdown
## R144 — 2026-05-12

### Added
- BM25 backfill resilience: skip docs with malformed embeddings (#234)
```

**Landing (`labyra-landing`):**
```markdown
## L3 — 2026-05-12

### Changed
- Pricing copy: "$149/lab/month" → "$129/lab/month" (positioning lock-in)
- Hero subhead now leads with "AI-native" instead of "AI-powered"
```

### Cross-repo references

When landing references app capabilities, link by app round number:

```markdown
<!-- In labyra-landing/CHANGELOG.md -->
## L7 — 2026-05-20
- Updated stats section to reflect [labbook-bku R150](https://github.com/emnam009/labbook-bku/releases/tag/r150) — Phase B.5 schema close
- Removed "beta" badge from Hero (app reached production-ready threshold)
```

When app references landing iterations (rare, only for marketing
launches that gate on app features):

```markdown
<!-- In labbook-bku/CHANGELOG.md -->
## R165 — 2026-08-15
- Multi-tenant rules namespace deployed
- Coordinated with [landing L18 launch announcement](https://github.com/emnam009/labyra-landing/releases/tag/l18)
```

### Frontmatter for landing changelog entries

`labyra-landing/src/content/changelog/lN-slug.md`:

```yaml
---
title: "Pricing lock-in (L3)"
date: 2026-05-12
iteration: L3
appRoundReference: R144  # optional — if change is triggered by app capability
summary: "Pricing copy update + hero rewording"
---
```

### When landing iteration and app round coincide

Sometimes a landing change is triggered by an app feature shipping. In
those cases, the landing CHANGELOG entry references the app round but
keeps its own L-number. Example:

- App ships R150 (Phase B.5 close, Unified Research Schema)
- Landing updates Features section to mention "Schema-aware search" → L7
- L7's frontmatter has `appRoundReference: R150`
- L7 is **not** renamed to "R150-L1" — that creates implicit coupling

The two repos can ship at different times: L7 might land 2 days before
or after R150 depending on copy review and app rollout windows.
